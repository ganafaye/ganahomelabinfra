Partie 1 — Concepts Fondamentaux

1.1

Virtualisation traditionnelle vs Conteneurs

Analyse des architectures : Hyperviseur Type 1, Type 2 et Conteneurs.

Figure 1 : Comparaison structurelle des couches d'abstraction

Question 1 : Analyse des différences structurelles et choix techniques.
  • a) Hyperviseur Type 2 : Il possède une couche supplémentaire car il s'exécute sur un Host OS. Contrairement au Type 1 qui accède directement au hardware, le Type 2 dépend du noyau de l'hôte pour gérer les ressources.
  • b) Absence de Guest OS : Les conteneurs partagent le noyau Linux de l'hôte. Ils isolent uniquement les processus et dépendances, ce qui supprime le besoin d'un OS complet par instance.
  • c) Usage Type 1 : Préféré pour une isolation matérielle totale ou pour faire tourner des OS hétérogènes (ex: Windows et Linux sur un même serveur physique).
1.2

Anatomie d'une plateforme de conteneurs

Figure 2 : Architecture générique et composants

Question 2 : Rôle des composants et gestion des images.

a) Le Container Engine est responsable du téléchargement (pull) de l'image depuis le Registry.

b) Le runtime haut niveau gère le cycle de vie (images, réseau), tandis que le bas niveau s'occupe de l'exécution via le noyau.

c) Les images sont en couches (layers) pour permettre la réutilisation et l'optimisation : une couche commune à plusieurs images n'est stockée qu'une seule fois.

1.3

Mécanismes d'isolation du noyau Linux

Figure 3 : Architecture interne d'exécution et isolation

Question 3 : Rôle de dockerd, containerd, runc et limites.

a) dockerd (API), containerd (Gestion) et runc (Exécution directe).

b) Les Regular Processes voient tout le système hôte ; les Containerized Processes sont isolés via les Namespaces.

c) Les Cgroups limitent la consommation de RAM et CPU.

d) L'isolation est moins forte car le noyau est partagé entre tous les conteneurs et l'hôte.

Isolation des Namespaces

Capture 6 : Comparaison 'ps aux' (Conteneur) vs 'docker top' (Hôte)
Question 8 : "Voir à travers" l'isolation

a) Dans le conteneur, seul le processus bash est visible (PID 1). Sur l'hôte, tous les processus système sont visibles.

b) C'est le mécanisme des Namespaces PID qui segmente la visibilité des processus.

c) La commande docker top (ou un ps aux | grep bash sur l'hôte) permet de voir le processus du conteneur car le noyau de l'hôte gère tous les processus, même isolés.

9.b) Protection : Les Cgroups agissent comme des gardes-fous. Si un conteneur dépasse sa limite de RAM, le noyau peut déclencher l'OOM Killer uniquement sur ce conteneur, protégeant ainsi la VM Ubuntu et les autres services.

Partie 4 — Standards OCI et Interopérabilité

4.1

Les Spécifications OCI

Analyse des standards de l'Open Container Initiative pour la portabilité des images et des runtimes.

Question 11 : Interopérabilité LXC et Docker

a) Apport des standards : Avant l'OCI, les formats étaient propriétaires. Grâce à l'Image Spec, une image créée par Docker peut être comprise par n'importe quel moteur compatible (comme Podman ou LXD).

b) LXC à partir de Docker : Oui, c'est possible. Le principe consiste à extraire le RootFS (le système de fichiers) de l'image Docker (via docker export ou des outils comme lxc-docker) pour l'utiliser comme base d'un conteneur système LXC. Cela permet de bénéficier de l'écosystème Docker Hub pour alimenter des conteneurs LXC.

Exercice — Inspection d'une image OCI

Manipulation des archives d'images pour comprendre la structure interne (Manifest, Layers).

Capture 9 : Extraction de l'archive alpine.tar et contenu du manifest.json
Question 12 : Analyse de l'image Alpine

a) Nombre de couches : L'image Alpine est extrêmement légère et ne contient généralement qu'une seule couche (layer) de base.

b) Rôle du SHA256 : Le champ sha256 sert d'identifiant unique et immuable pour chaque couche et pour la configuration globale. C'est le "Content Addressable Storage".

c) Garantie technique : Cela garantit l'intégrité des données. Lors du téléchargement, Docker recalcule le hash. Si le hash calculé ne correspond pas au SHA256 du manifeste, le téléchargement est rejeté, protégeant ainsi contre les corruptions de fichiers ou les attaques de type "Man-in-the-middle".
