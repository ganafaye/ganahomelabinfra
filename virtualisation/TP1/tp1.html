Ma Réponse :
Mon processeur est un [À COMPLÉTER, ex: Intel Core i7-1165G7]. Le flag vmx est présent, confirmant le support d'Intel VT-x.

Question Q1.2 : Expliquez le rôle du Domaine 0 (Dom0) par rapport aux Domaines n (DomU). Pourquoi le Dom0 est-il dit « privilégié » ?
Ma Réponse :
Le Dom0 est la seule machine virtuelle lancée au démarrage du système ayant un accès direct au matériel. Son rôle est de gérer les autres machines virtuelles (les DomU). Il est dit "privilégié" car il possède les pilotes de périphériques et gère l'allocation des ressources.

Risque sécurité : Si le Dom0 est compromis, l'attaquant prend le contrôle total de l'hyperviseur et de toutes les machines virtuelles résidentes.

Capture 2 : Exécution Java et inspection des Magic Bytes
Question Q1.3 : Que signifient les bytes 'CA FE BA BE' ? Pourquoi dit-on que la JVM implémente le principe WORA ?
Ma Réponse :
Les bytes CA FE BA BE sont la signature magique identifiant un fichier comme étant du bytecode Java.

Contrairement à un exécutable C (natif et lié à une architecture CPU spécifique), le bytecode Java est universel. La JVM (propre à chaque OS) se charge de traduire ce bytecode en instructions machine en temps réel. C'est ce qui permet le WORA : on écrit le code une fois, et il tourne sur n'importe quel système possédant une JVM.

Question Q2.1 : Comparez la RAM (free -h) avec la RAM consommée sur l'hôte. Expliquez le « memory balloon driver ».
Ma Réponse :
La RAM vue par la VM correspond à l'allocation fixe (1 Go), tandis que l'hôte ne consomme réellement que ce dont la VM a besoin à l'instant T.

Le Memory Balloon Driver est un pilote qui permet à l'hyperviseur de "gonfler" un ballon invisible dans la RAM de l'invité pour récupérer les pages mémoire inutilisées et les restituer à l'OS hôte (ThinkPad), optimisant ainsi la mémoire physique globale.
Question Q2.2 : Citez 2 mécanismes permettant à un OS invité de détecter qu'il est dans une VM.
Ma Réponse :
1. L'instruction CPUID : L'OS interroge le processeur ; l'hyperviseur renvoie souvent une signature spécifique (ex: "Oracle" ou "KVMKVMKVM").
2. Adresses MAC & Artefacts : La présence de périphériques virtuels spécifiques (ex: contrôleur graphique VBoxVGA) ou d'adresses MAC réservées aux constructeurs virtuels.

Note : En cybersécurité (analyse de malwares), on utilise l'anti-VM pour cacher ces indices et tromper le logiciel malveillant.

Question Q2.4 : Architecture réseau et rôle du bridge.
Ma Réponse :
[ LAN (Box/Router) ] <---> [ Carte Physique (WiFi/Eth) ]
                                   |
                          [ vSwitch / Bridge VirtualBox ]
                               /            \
                    [ Hôte Ubuntu ]      [ VM Ubuntu Server ]
                    (IP: 192.168.1.X)    (IP: 192.168.1.Y)
Le Bridge agit comme un Switch physique virtuel : il intercepte les paquets sur la carte réseau physique et les redirige vers la VM en fonction de son adresse MAC virtuelle. Cela permet à la VM d'être un nœud complet du réseau.

Question Q3.1 : Expliquez le mécanisme de snapshot (copy-on-write, fichiers .vdi delta). Quelle différence avec une sauvegarde complète ?
Ma Réponse :
Le snapshot utilise le mécanisme Copy-on-Write (CoW) : le fichier disque original (.vdi) devient "lecture seule" et VirtualBox crée un nouveau fichier VDI Delta. Toutes les nouvelles écritures sont redirigées vers ce delta.

Différence : Un snapshot est instantané et ne stocke que les modifications, alors qu'une sauvegarde complète duplique l'intégralité du disque (beaucoup plus long et lourd).
Limites : Un snapshot ne protège pas contre la panne physique du disque dur (si le SSD du ThinkPad lâche, le snapshot est perdu) ou la corruption du fichier source.

Question Q3.2 : Différence entre clone lié et clone complet ? Lequel choisir pour 20 environnements de test ?
Ma Réponse :
Un clone complet est une copie 100% indépendante. Un clone lié dépend de la VM source et ne stocke que les fichiers modifiés par rapport à celle-ci.

Pour 20 environnements, je choisirais le clone lié car il permet d'économiser énormément d'espace disque (quelques Mo par clone au lieu de 20 Go par VM). Le risque principal est la dépendance : si la VM parente est supprimée, les 20 clones deviennent inutilisables.

Capture 8 : Inspection du descripteur XML de l'appliance OVF
Question Q3.3 : Que contient le fichier .ovf ? Pourquoi est-ce un standard important ? Situations utiles ?
Ma Réponse :
Le fichier .ovf est un fichier XML contenant les métadonnées de la VM : configuration CPU, RAM, disques virtuels, et matériel requis.

Ce standard est crucial pour éviter l'enfermement propriétaire (vendor lock-in).
Situations utiles :
1. Migrer une VM de VirtualBox vers un serveur VMware ESXi en entreprise.
2. Distribuer une application pré-installée (Appliance) à des clients.
3. Archiver une VM dans un format lisible sur le long terme.

Question Q4.1 : Pourquoi choisir Proxmox en production plutôt que VirtualBox ? (5 raisons)
Ma Réponse :
1. Architecture Bare-Metal : Performances supérieures car pas d'OS hôte intermédiaire.
2. Haute Disponibilité (HA) : Redémarrage automatique des VMs sur un autre nœud en cas de panne physique.
3. Gestion Centralisée : Administration de centaines de serveurs via une seule interface web.
4. Sauvegardes Natives : Planification de backups à chaud sans outils tiers.
5. Multi-Tenancy : Gestion précise des droits pour différents administrateurs ou clients.

Résumé de cours : Ce qu'il faut retenir

Les 3 piliers de la Virtualisation
  • Isolation : Chaque VM est un "bac à sable". Une panne ou un virus dans une VM n'affecte pas l'hôte.
  • Encapsulation : Une VM est un simple fichier (VDI, VMDK). On peut la copier, la déplacer ou la supprimer comme un document.
  • Indépendance matérielle : Le matériel vu par l'OS invité est virtuel (générique), ce qui permet de migrer une VM entre deux serveurs physiques différents.
Concepts Clés manipulés
  • Hyperviseur : Le logiciel "chef d'orchestre" qui distribue les ressources CPU/RAM.
  • Snapshot : Sauvegarde de l'état des registres et du disque à un instant T (technique Copy-on-Write).
  • Standard OVF : Le format XML universel pour éviter d'être bloqué chez un seul constructeur (Interopérabilité).
